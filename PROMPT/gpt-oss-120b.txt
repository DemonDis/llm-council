**Тема:** Анализ фреймворков для промпт‑инжиниринга
**Подготовил:** Системный аналитик с 10‑летним стажем в ИТ
**Дата:** 19 января 2026 г.

*В подготовке ответа использован внутренний процесс «ПОНИМАНИЕ → КОНТЕКСТ → ПЕРСПЕКТИВЫ → РАССУЖДЕНИЕ → СИНТЕЗ → ОГРАНИЧЕНИЯ», что обеспечивает полноту, актуальность и структурность материала.*

---

## 1. Введение

Промпт‑инжиниринг – ключевая компетенция при работе с LLM (Large Language Models). Унифицированные фреймворки упрощают формулирование запросов, повышая предсказуемость и качество выдачи. Цель отчёта – сравнить семь популярных фреймворков, выявить их сильные и слабые стороны и сформировать рекомендации по применению в разных проектах ИТ‑подразделения.

---

## 2. Основная часть

### 2.1 Краткие описания фреймворков

| № | Фреймворк | Расшифровка | Суть структуры |
|---|-----------|-------------|----------------|
| 1 | **RTF** | Role, Task, Format | Описывает роль LLM, конкретную задачу и требуемый формат ответа. |
| 2 | **COSTAR** | Context, Objective, Style, Tone, Audience, Response | Полностью охватывает контекст и желаемый стиль коммуникации. |
| 3 | **IEEI** | Information, Example, Explanation, Instruction | Фокусируется на предоставлении данных, примеров и инструкций. |
| 4 | **APE** | Action, Purpose, Expectation | Сокращённый набор, акцентирует действие, цель и ожидаемый результат. |
| 5 | **PECRA** | Purpose, Expectation, Context, Request, Audience | Расширенный вариант APE с отдельным пунктом «Запрос» и аудиторием. |
| 6 | **OSCAR** | Objective, Situation, Context, Action, Result | Подходит для сценариев, где важна последовательность событий и результат. |
| 7 | **TAG** | Task, Action, Goal | Минималистичный набор, удобен для быстрых запросов. |

---

### 2.2 Правила формирования промптов

| Фреймворк | Ключевые правила |
|-----------|------------------|
| **RTF** | 1. Чётко укажите роль («ты — технический писатель»). 2. Формулируйте задачу в виде глагола («сгенерировать таблицу»). 3. Определите требуемый формат (JSON, Markdown, список). |
| **COSTAR** | 1. Описать контекст (текущий проект, ограничения). 2. Сформулировать цель. 3. Указать стиль (деловой, разговорный). 4. Тон (нейтральный, мотивирующий). 5. Аудиторию (разработчики, менеджеры). 6. Ожидаемый тип ответа (пример кода, анализ). |
| **IEEI** | 1. Предоставьте всю релевантную информацию. 2. Приведите пример требуемого вывода. 3. Поясните, почему пример важен. 4. Дайте чёткие инструкции по выполнению. |
| **APE** | 1. Определите действие, которое должен выполнить LLM. 2. Укажите цель действия. 3. Описать ожидаемый результат (точность, длина). |
| **PECRA** | 1. Сформулируйте цель. 2. Укажите требования к результату. 3. Описать контекст (данные, ограничения). 4. Сформулировать конкретный запрос. 5. Указать аудиторию. |
| **OSCAR** | 1. Определить объект цели. 2. Описать текущую ситуацию. 3. Дать контекстные сведения. 4. Указать требуемое действие. 5. Ожидаемый результат. |
| **TAG** | 1. Сформулировать задачу. 2. Описать действие. 3. Указать цель/результат. Минимум слов, максимальная ёмкость. |

---

### 2.3 Плюсы и минусы

| Фреймворк | Плюсы | Минусы |
|-----------|-------|--------|
| **RTF** | • Простота и быстрота применения.<br>• Хорошо подходит для технической документации.<br>• Чёткое разделение ролей и формата. | • Ограниченный набор параметров (нет стиля/тона).<br>• Не учитывает аудиторию. |
| **COSTAR** | • Полнота: покрывает почти все аспекты коммуникации.<br>• Удобен для кросс‑функциональных запросов.<br>• Позволяет управлять качеством тона и стиля. | • Длинные запросы → рискует превысить токен‑лимит.<br>• Требует более высоких навыков написания. |
| **IEEI** | • Информативность: данные + пример повышают точность.<br>• Явные инструкции снижают неоднозначность.<br>• Подходит для обучающих и рефакторинговых задач. | • Ориентирован на небольшие контексты.<br>• При большом объёме информации становится громоздким. |
| **APE** | • Краткость, экономия токенов.<br>• Фокус на конечный результат.<br>• Идеален для автоматических скриптов. | • Не описывает контекст и аудиторию.<br>• Сложно задать стилистические требования. |
| **PECRA** | • Сбалансированное сочетание цели, контекста и аудитории.<br>• Чёткое разделение «запроса» повышает контроль.<br>• Хорошо подходит для клиент‑ориентированных сервисов. | • Всё же требует более детального описания, чем APE.<br>• При плохом формулировании «Expectation» может вести к разночтениям. |
| **OSCAR** | • Поддерживает повествовательный подход (ситуация → действие → результат).<br>• Отлично подходит для сценариев бизнес‑анализа и кейс‑стади. | • Не всегда удобно для чисто технических запросов.<br>• Перегружен в случаях простых задач. |
| **TAG** | • Самый короткий набор – экономия токенов.<br>• Идеален для быстрого прототипирования и CI‑тестов. | • Недостаточно контекстного уточнения.<br>• Риск неверной интерпретации без стиля/тона. |

### 2.4 Примеры промптов

| Фреймворк | Пример запроса (англ.) |
|-----------|------------------------|
| **RTF** | **Role:** You are a senior backend engineer.<br>**Task:** Generate a PostgreSQL schema for a multi‑tenant SaaS product.<br>**Format:** Provide the output as a Markdown code block. |
| **COSTAR** | **Context:** Our micro‑service receives JSON payloads from an IoT platform.<br>**Objective:** Write a validation function.<br>**Style:** Technical, concise.<br>**Tone:** Neutral.<br>**Audience:** Junior developers.<br>**Response:** Provide Python code with comments. |
| **IEEI** | **Information:** The system stores user sessions in Redis with a TTL of 30 min.<br>**Example:** `SET session:id value EX 1800`.<br>**Explanation:** We need a function that refreshes the TTL without overwriting the value.<br>**Instruction:** Write a Go routine that performs this operation safely. |
| **APE** | **Action:** Create a unit test suite.<br>**Purpose:** Ensure 90 % coverage of the authentication module.<br>**Expectation:** Tests must be runnable with `go test ./...` and include edge‑case scenarios. |
| **PECRA** | **Purpose:** Automate generation of release notes.<br>**Expectation:** Output in Markdown, grouped by feature, bugfix, and chore.<br>**Context:** Tags follow `vX.Y.Z` convention.<br>**Request:** Pull commit messages from the last tag to HEAD.<br>**Audience:** Product managers. |
| **OSCAR** | **Objective:** Reduce API latency.<br>**Situation:** Current average response time is 250 ms.<br>**Context:** Service runs on AWS Lambda with cold‑start issues.<br>**Action:** Propose code‑level optimizations and infrastructure tweaks.<br>**Result:** Provide a 3‑step plan with expected latency reduction percentages. |
| **TAG** | **Task:** Summarize the attached RFC.<br>**Action:** Extract key decisions.<br>**Goal:** Produce a 5‑sentence executive summary. |

### 2.5 Сравнительная таблица характеристик

| Фреймворк | Степень детализации | Токен‑эффективность | Поддержка стилистики | Ориентированность на аудиторию | Идеальные сценарии |
|-----------|---------------------|----------------------|----------------------|-------------------------------|--------------------|
| **RTF** | Средняя | Высокая | ❌ | ❌ | Техническая документация, генерация кода |
| **COSTAR** | Высокая | Низкая | ✅ | ✅ | Кросс‑функциональные запросы, маркетинг, клиент‑фронт |
| **IEEI** | Средняя‑высокая | Средняя | ✅ (через пример) | ❌ | Обучающие материалы, рефакторинг |
| **APE** | Низкая | Очень высокая | ❌ | ❌ | Автоматизация, CI‑скрипты |
| **PECRA** | Высокая | Средняя | ✅ | ✅ | Сервисные запросы, клиент‑ориентированные ответы |
| **OSCAR** | Высокая | Низкая‑средняя | ✅ | ✅ | Бизнес‑аналитика, кейс‑стади |
| **TAG** | Низкая | Очень высокая | ❌ | ❌ | Быстрые прототипы, проверка гипотез |

## 3. Выводы и рекомендации

1. **COSTAR** – наиболее универсальный, но требует контроля за объёмом токенов. Рекомендуется использовать в проектах, где важны **tone**, **style** и **аудитория** (например, генерация клиентской коммуникации, обучающих материалов, PR‑контент).

2. **RTF** – лучший выбор для **технических** задач, где нужен строгий формат вывода (спецификации, API‑документация). Его простота ускоряет разработку запросов в CI‑конвейерах.

3. **IEEI** демонстрирует высокую точность при **обучающих** и **рефакторинговых** сценариях, где пример‑ориентированность повышает качество результата.

4. **APE** и **TAG** – «lite‑варианты» для **быстрого прототипирования**, автоматизации тестов и скриптов. Их экономичность позволяет использовать их в **high‑throughput** средах (batch‑generation, micro‑service calls).

5. **PECRA** сочетает преимущества APE и COSTAR, предоставляя **баланс** между детализацией и эффективностью. Идеален для **служебных** запросов (release‑notes, SLA‑формулировки).

6. **OSCAR** – оптимален для **бизнес‑аналитики** и описания процессов, где требуется показать причинно‑следственную связь (проблема → действие → результат).

### Приоритеты внедрения

| Приоритет | Фреймворк | Целевое подразделение |
|-----------|-----------|------------------------|
| 1 | **COSTAR** | Маркетинг, поддержка клиентов, HR‑коммуникации |
| 2 | **RTF** | Разработка (API‑документация, генерация кода) |
| 3 | **PECRA** | Продуктовый менеджмент, релиз‑процессы |
| 4 | **IEEI** | Обучение, внутренние вики, документация процессов |
| 5 | **OSCAR** | Бизнес‑аналитика, стратегическое планирование |
| 6 | **APE** / **TAG** | CI/CD, автоматические тесты, быстрая генерация контента |

**Итог:** Фреймворки различаются степенью детализации, токен‑экономичностью и фокусом на стиль/аудиторию. При выборе следует ориентироваться на конкретный контекст задачи: для технической генерации – RTF; для клиент‑ориентированных коммуникаций – COSTAR; для быстрых скриптов – APE/TAG; для аналитических сценариев – OSCAR. При правильном их применении можно существенно повысить качество и предсказуемость ответов LLM, сократить количество итераций и ускорить процессы разработки в ИТ‑подразделении.
